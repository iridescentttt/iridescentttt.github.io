分布式系统 | docker 迁移与恢复

No tag
Publish Date:   2022-09-27

实验题：CRIU是一种在用户空间实现的进程或者容器checkpoi nt 和restore的方法， 从而实现进程或者容器的保存和恢复。 请利用CRIU实现进程和容器的迁移，并利用样例程序如Web程 序等测试迁移过程中的性能损耗（如延迟变化或者请求错误率 等）、观察发现，并撰写报告, 报告的模板可以在课程网站找到。

解答:

使用工具

本次实验采用 multipass+docker+nginx+locust 进行具体实现. 它们分别作用如下所示.

  • multipass 构建虚拟机环境. 一个是生成 checkpoint 的环境 linn, 另一个是恢复 checkpoint 的环境 test.
  • docker 用于构建容器.
  • nginx 提供反向代理, 方便本地请求错误率的记录.
  • locust 是 python 实现的一个压测工具, 用于记录请求错误率.

虚拟机环境

首先使用 multipass 构建两个虚拟机环境. 各自 ip 如下所示.

可以看到两个虚拟机环境的 ip 分别是 192.168.64.4 和 192.168.64.6. multipass 中这些虚拟机可以通过上述局域网 ip 和宿主机进行相互通信.

为了实现通信, 我们给 test 实例配置 ssh, 并把密码也设置成为 test. 之后就可以使用 scp 命令进行 checkpoint 文件的传输.

Docker 和 criu 安装

然后我们按照教程进行 criu 的安装配置. 

sudo apt update
sudo apt upgrade -y
sudo apt install -y build-essential
sudo apt install -y pkg-config
sudo apt install -y libnet-dev python3-yaml libaio-dev libnftables-dev
sudo apt install -y libprotobuf-dev libprotobuf-c-dev protobuf-c-compiler protobuf-compiler python3-protobuf libnl-3-dev libcap-dev python3-future

# criu install
git clone https://github.com/checkpoint-restore/criu.git
cd criu
sudo make clean
sudo make
sudo cp ./criu/criu /usr/local/bin

安装 docker.

wget https://download.docker.com/linux/ubuntu/dists/xenial/pool/stable/arm64/docker-ce_17.09.0~ce-0~ubuntu_arm64.deb
sudo dpkg -i docker-ce_17.09.0~ce-0~ubuntu_arm64.deb
# Daemon environment will run automaticly

# Run docker without sudo
sudo groupadd docker
sudo usermod -aG docker $USER

检验 criu 是否成功安装.

检验 docker 是否成功安装.

使用 docker 启动 nginx 网络服务

直接 docker pull 下来 nginx, 然后将容器的 80 端口到 linn 服务器上的 8080 端口即可. 

docker pull nginx
docker run --name nginx -p 8080:80 -d nginx

然后可以检查 docker 容器状态.

此时通过本地访问 linn 服务器 ip 上 8080 端口即可检验 nginx 成功启动.

在 linn 服务器上创建脚本保存 checkpoint 并传递到 test 服务器

在 host 机器 linn 上使用下列脚本 checkpoint.sh 来进行创建 checkpoint 以及文件的传输.

checkpoint.sh 

# 错误继续执行
set +e

# 保存镜像
docker save nginx > nginx.tar
echo "保存镜像成功"

# 删除之前保存的 ckp1, 重新创建 checkpoint
if [[ -n $(docker checkpoint ls nginx | grep ckp1) ]]; then
       docker checkpoint rm nginx ckp1 >/dev/null 2>&1
fi

# 启动容器
docker start $(docker ps -aqf"name=nginx") >/dev/null 2>&1
echo "启动容器成功"

# 创建 checkpoint
docker checkpoint create nginx ckp1 >/dev/null 2>&1
echo "创建 checkpoint 成功"

# 使用 scp 传输镜像文件以及 checkpoint 文件夹
sudo scp nginx.tar ubuntu@192.168.64.6:~/nginx.tar >/dev/null 2>&1
sudo scp -r /var/lib/docker/containers/$(docker ps  --no-trunc -aqf"name=nginx")/checkpoints/ckp1 ubuntu@192.168.64.6:~ >/dev/null 2>&1
echo "传输成功"

set -e

注意到这里将 checkpoint 保存到的是 docker 容器的默认文件夹中, 在实际测试过程中, 如果保存到自定义的文件夹中好像会导致文件权限检验失败而不能在 test 机器上成功 restore. 一开始我以为是 docker 版本不兼容的问题, 重装了好多遍 docker 后来才发现是文件权限检验的问题.

如下图所示, 查看 log 提示说明, 发现可以使用 –skip-file-rwx-check 选项来跳过 check 操作. 但因为 criu 内嵌到 docker start –checkpoint 命令中, 所以不清楚怎么通过加选项的方式解决.

最终传输的文件为镜像文件 nginx.tar 以及 checkpoint 文件夹 ckp1, 传递的目的地都是主目录下.

在 test 服务器中复原 checkpoint

在目标服务器 test 上使用 restore.sh 脚本进行 checkpoint 的复原. 这里依然使用的是 8080 端口进行映射. 

"restore.sh" 23L, 663B                                                                                                                                 23,6          All
# 错误继续执行
set +e

# 暂停 nginx_test 容器的运行, 之后删除该容器
docker stop $(docker ps -aqf"name=nginx_test") >/dev/null 2>&1
docker rm $(docker ps -aqf"name=nginx_test") >/dev/null 2>&1
echo "删除容器成功"

# 删除存在的 nginx 镜像
docker image rm nginx >/dev/null 2>&1
echo "删除镜像成功"

# 载入镜像
docker load -i ~/nginx.tar

# 创建 docker image
docker create -p 8080:80 --name nginx_test --security-opt seccomp:unconfined nginx

# 载入 checkpoint
sudo cp -r ~/ckp1 /var/lib/docker/containers/$(docker ps  --no-trunc -aqf"name=nginx_test")/checkpoints/ckp1
docker start --checkpoint=ckp1 nginx_test

set -e

然后可以检查 docker 容器状态.

此时通过本地访问 test 服务器 ip 上 8080 端口即可检验 nginx 成功启动.

本地 nginx 反向代理

为了方便后续进行请求错误率的记录, 在本地进行反向代理.

在当前文件夹下创建 nginx.conf 来进行反向代理.

nginx.conf

events {
    worker_connections 1024;
}

http {
    server {
        listen          8000;
        server_name     localhost;

        location / {
            proxy_pass http://192.168.64.6:8080;
        }
    }
}

该文件配置 nginx 将本地的 8000 端口映射到 linn 服务器的 8080 端口.

采用下列指令运行 nginx.

nginx -c $(pwd)/nginx.conf

然后通过本地 8000 端口进行检验.

总迁移脚本

可以在本地结合 multipass exec 指令来实现总的迁移脚本 run.sh. 

# mac 下修改 nginx.conf
sed -i '' 's/192.168.64.6/192.168.64.4/g' nginx.conf
pkill nginx

multipass exec linn bash checkpoint.sh 
multipass exec test bash restore.sh

# mac 下修改 nginx.conf
sed -i '' 's/192.168.64.4/192.168.64.6/g' nginx.conf
nginx -c $(pwd)/nginx.conf

lucust 压测

在本地创建 lucustfile.py 配置文件, 方便 lucust 对 127.0.0.1:8000 进行接口测试.

from locust import HttpUser, task, between

# 这个类是我们用来模拟发起请求的用户
# 此类继承自 HttpUser, HttpUser 类包含的 client 属性就是一个用来发 Http 请求的 HttpSession
# 当开始运行测试，locust 会给每个模拟的 user 创建一个此类的实例
class QuickstartUser(HttpUser):
    # wait_time 定义了模拟用户的请求间隔时间，这里的含义是每个 user 会在一个请求结束后随机等 1-5s	
    wait_time = between(1, 5)

    # task 装饰架定义了模拟用户的行为，即发送什么样的 http 请求
    # 每个 user 都会起一个 greenlet 执行这些方法
    @task
    def hello_world(self):
        self.client.get("http://127.0.0.1:8000/")

登录 127.0.0.1:8089 进入控制台, 设置最多并发 100 个用户, 每秒启动 100 个虚拟用户, 持续 60s.

压测结果中, 请求失败率为 0%.

然后重新进行压测并执行迁移脚本, 请求错误率为 11%.

观察具体数据, 可以发现在迁移过程出现了短暂的请求失败, 迁移完成后成功恢复该接口服务.

思考

这次是在 multipass 模拟的局域网中进行迁移, scp 时间较短, 而且 nginx 镜像文件也较小. 如果是在互联网上迁移正式的服务的话, 迁移时间肯定要远远长于本次实验时间.
